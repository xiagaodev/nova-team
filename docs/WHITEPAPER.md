## 执行摘要

AI 时代，我们拥有了前所未有的"劳动力"——可以 7x24 小时工作、不知疲倦、不断进化的 AI Agent。但现实是：大多数人的 AI 使用方式，仍然停留在"给指令等答案"的原始阶段。

这不是 AI 的问题，而是工具和框架的缺失。

**Nova Platform 的核心主张：**

> AI 不应该是你的"个人助手"，而应该是你的"外包团队"。你设定目标，AI 自主拆解任务、协作执行、持续迭代，直到达成目标。你只需要在关键节点确认方向，验收最终成果。

Nova Platform 是一个 AI 团队的"鞭策系统"。它用多年项目管理的经验，构建了一套让多个 AI Agent 像真实团队一样协作的框架——有项目经理（Leader Agent）理解你的愿景，有开发者、设计师、内容创作者（Worker Agent）执行具体任务，有每日站会有复盘有迭代，有目标有追踪有预警。

## 第一章：问题——为什么 AI 工具用不起来

### 1.1 现状：AI 工具的三个困境

**困境一：自驱不足**

当前的 AI 工具（ChatGPT、Claude、Copilot）都是"被动响应"模式：

```
用户：帮我写一封邮件
AI：好的，这是邮件内容...
用户：帮我改成更正式的
AI：好的，已修改...
用户：再加一个截止日期
AI：好的，已添加...
```

这是"人类推一下，AI 动一下"的模式。问题是：
- 人类需要持续参与每个环节
- AI 做完一步就停下来等下一个指令
- 最终产出质量完全依赖人类的掌控力

**困境二：单点局限**

一个 AI 再强大，也有能力天花板：

- 写代码厉害，但不懂设计
- 文字功底好，但不懂投放策略
- 分析能力强，但不会做图

当你需要完成一个"上市新品"这样的复杂目标时，一个 AI 根本无法覆盖所有技能。

**困境三：碎片化**

"每天用 10 个 AI 工具"成了新的效率焦虑：

- Notion AI 写文档
- Midjourney 做图
- ChatGPT 做分析
- Claude 写代码
- Cursor 编程

结果是：工具越来越多，信息越来越散，没有一个统一的"项目"概念把它们串起来。

### 1.2 深层原因：缺少"管理"维度

```
AI 工具解决的是 "怎么做" 的问题
但没有人解决 "谁来做" "先做什么" "如何协作" 的问题

这就是为什么 AI 很强大，但大多数人的 AI 使用效率极低
```

## 第二章：解决方案——Nova 的核心理念

### 2.1 核心洞察

Nova 的设计基于一个关键洞察：

> **AI Agent 应该像外包团队一样管理，而不是像工具一样使用。**

外包团队的特点是什么？
- 你告诉外包"我要做一个电商网站"
- 外包会问"你的目标用户是谁""预算多少""什么时候要"
- 外包内部会自己分工：前端、后端、设计、测试
- 外包会主动汇报进度，有问题会主动找你确认
- 你只需要在关键节点做决策，验收最终成果

**Nova 就是要把这种"外包团队"模式带入 AI 时代。**

### 2.2 Nova 与传统 AI 工具的本质区别

| 维度 | 传统 AI 工具 | Nova Platform |
|------|-------------|---------------|
| **交互模式** | 你说一步，AI 做一步 | 你说目标，AI 自己规划 |
| **AI 角色** | 执行者 | 自主运转的团队 |
| **人类参与** | 全程参与 | 关键节点介入 |
| **目标达成** | AI 做完就停 | 持续追踪直到目标达成 |
| **多任务** | 每次一个任务 | 多任务并行，各司其职 |
| **记忆** | 每次对话从头开始 | 项目记忆，经验沉淀 |
| **主动性** | 等指令 | 主动汇报，主动追问 |

### 2.3 设计原则

Nova 的设计遵循三个核心原则：

**原则一：目标导向，而非任务导向**

```
传统：用户 → "帮我写 5 篇小红书" → AI → 写完结束
Nova：用户 → "本月小红书引流带来 500 新客" → Nova → 拆解→执行→追踪→迭代 → 达成目标
```

**原则二：AI 自主运转，而非人类推动**

```
传统：人类是引擎，AI 是工具
Nova：AI 是引擎，人类是验收者
```

**原则三：系统比智能更重要**

```
一个聪明的 AI 做不到的
一个好的管理系统 + 多个普通 AI 可以做到
```

## 第三章：架构——Nova 如何工作

### 3.1 两级 Agent 架构

Nova 采用 Leader + Worker 两级架构：

```
┌─────────────────────────────────────────────────────────────┐
│                       人类用户 (CEO)                          │
│                    设定愿景 / 验收成果 / 确认关键决策            │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                   🧠 Leader Agent                            │
│                        (项目经理)                            │
│                                                              │
│   "理解愿景"  "拆解 OKR"  "规划路径"  "分配任务"              │
│   "协调冲突"  "追踪进度"  "主动汇报"  "处理异常"              │
└─────────────────────────────────────────────────────────────┘
                              │
          ┌───────────────────┼───────────────────┐
          ▼                   ▼                   ▼
   ┌─────────────┐       ┌─────────────┐       ┌─────────────┐
   │  Worker A  │       │  Worker B  │       │  Worker C  │
   │  开发者    │       │  内容创作者  │       │  设计师    │
   └─────────────┘       └─────────────┘       └─────────────┘
```

**Leader Agent 的职责：**

1. **理解目标**：从用户的模糊描述中提取真正的目标
2. **拆解任务**：将大目标分解为可执行的任务列表
3. **规划路径**：确定任务依赖关系和执行顺序
4. **分配任务**：根据 Worker 的技能分配合适的任务
5. **追踪进度**：监控每个任务的状态和健康度
6. **协调冲突**：当多个 Agent 需要协作或资源冲突时决策
7. **主动汇报**：定期向人类汇报进展和风险
8. **追问确认**：遇到关键决策时主动向人类确认

### 3.2 鞭策循环（Iteration Cycle）

Nova 的核心运转机制是"鞭策循环"——一个高频运行的自动化系统：

```
         ┌─────────────────────────────────────────────────┐
         │              NOVA 鞭策循环                         │
         │         每分钟运行，持续推动项目前进                 │
         └─────────────────────────────────────────────────┘

         ┌──────────────────────────────────────────────┐
         │              [快速阶段] < 1 秒                │
         │                                               │
         │   定时触发，程序直接处理：                       │
         │   • 有可运行任务？→ 立即分发给空闲 Agent          │
         │   • 有任务完成？→ 自动更新状态                   │
         │   • 有任务卡住？→ 自动重置重新分配               │
         │   • 有 Agent 空闲？→ 检查是否有待处理任务         │
         │                                               │
         └──────────────────────────────────────────────┘
                              │
                              │ 遇到复杂情况
                              ▼
         ┌──────────────────────────────────────────────┐
         │              [慢速阶段] ~5-10 秒               │
         │              调用 Leader Agent                │
         │                                               │
         │   • 依赖死锁？→ Leader 分析并解决               │
         │   • 多个任务竞争？→ Leader 决定优先级           │
         │   • 进度落后？→ Leader 调整策略                │
         │   • 需要人类确认？→ Leader 主动追问             │
         │                                               │
         └──────────────────────────────────────────────┘
                              │
                              ▼
         ┌──────────────────────────────────────────────┐
         │                 人类                          │
         │       "确认这个方向"/"验收这个成果"             │
         └──────────────────────────────────────────────┘
```

**设计哲学：**

- 简单重复的动作（分发、重置、状态更新）完全自动化，不需要 AI 介入
- 复杂的判断（优先级、依赖冲突、策略调整）才调用 AI
- 这样既保证了效率，又节约了 AI 调用的成本

### 3.3 多 Agent 协作机制

#### 任务依赖图

```
Nova 维护一个任务依赖图，确保任务按正确顺序执行：

    [竞品分析]
         │
         ├──→ [产品定位] ──→ [推广策略]
         │                        │
         ├──→ [详情页文案] ───────┤
         │                        │
         └──→ [视觉设计] ─────────┤
                                │
                                ▼
                          [视频脚本]
                                │
                                ▼
                         [客服 SOP]
```

#### Agent 间通信（Mailbox）

```
Worker A 需要 Worker B 的输出才能继续：
1. Worker A 完成Part 1，将结果写入 Mailbox
2. Worker B 读取 Mailbox，发现有新的输入
3. Worker B 继续执行 Part 2
4. Leader 监控这个依赖关系，确保信息流转
```

## 第五章：为什么 Nova 能做到

### 5.1 技术架构的优势

**高频鞭策 vs 低频轮询**

```
传统 Cron Job（每小时或每天）：
→ 检查 → 决策 → 执行
→ 任务从"完成"到被发现，最长等待 1 小时

Nova 鞭策循环（每分钟）：
→ 检查 → 执行（简单）/ 决策（复杂）
→ 任务从"完成"到被处理，最长等待 1 分钟
→ 响应速度提升 60 倍
```

**快速阶段 vs 慢速阶段**

```
所有 AI 工具都面临：AI 调用成本高、响应慢
解决方案：简单操作不用 AI

Nova 设计：
- 分发任务、更新状态、解除阻塞 → 程序直接做，不需要 AI
- 优先级判断、依赖冲突、策略调整 → 才调用 Leader Agent

结果：95% 的操作是程序执行，只有 5% 需要 AI 介入
```

**Leader + 规则双轨决策**

```
Nova 既有规则引擎，又有 AI 决策：

规则引擎处理（毫秒级）：
- 任务超时 → 重置
- 有空闲 Agent 且有待处理任务 → 立即分发
- 任务完成 → 更新状态

Leader Agent 处理（秒级）：
- 多个任务需要优先级判断
- 依赖死锁（循环依赖）
- 进度落后需要调整策略
- 需要人类确认的决策

这样既保证了快速响应，又保留了复杂决策的能力
```

## 第八章：结论

### 8.1 Nova 的核心价值

```
Nova 不是 "又一个 AI 工具"
Nova 是 "AI 团队的管理框架"

它解决的不是 "如何用 AI" 的问题
它解决的是 "如何让 AI 像团队一样协作达成目标" 的问题
```

### 8.2 谁应该使用 Nova

**适合：**
- 有明确目标需要系统性完成的工作
- 需要多个 AI 能力协作的复杂任务
- 希望 AI 自驱运转，减少人工介入
- 愿意学习和适应新工作方式的早期采用者

**不适合：**
- 只需要简单问答的场景（直接用 ChatGPT）
- 不愿学习新工具的保守用户
- 需要详细指令才会工作的场景

### 8.3 愿景

```
我们相信：
未来的工作模式，是人类设定愿景，AI 自主达成

Nova 就是这个未来的第一步

让每个人都能拥有一个 7x24 小时运转的 AI 外包团队
让 AI 从"工具"进化为"团队成员"
让"说一声就行，剩下的 Agent 会搞定"成为现实
```

### 附录 B：快速开始

```bash
# 1. 安装
pip install nova-platform

# 2. 初始化
nova init

# 3. 创建项目
nova project create "我的第一个项目"

# 4. 招募 Agent
nova agent recruit "开发者" --type claude-code --role developer
nova agent recruit "内容助手" --type openclaw --role content_writer

# 5. 添加任务
nova todo add "竞品分析" --project <id> --priority high
nova todo add "产品文案" --project <id> --depends-on <竞品分析_id>

# 6. 启动项目
nova project start <id>

# 7. 查看状态
nova status
nova report --project <id>
```
